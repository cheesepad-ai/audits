#### üü° [M-1] Trading Lock Can Block All Transfers Until Owner Enables Trading

**Description:**
All transfers require `tradingActive == true` or either party whitelisted via `_isExcludedFromTradingLock`. Before `tradingActive` is enabled, non-whitelisted addresses cannot transfer. If liquidity is added prematurely without proper whitelisting (e.g., pair/router), buys/sells will revert, effectively blocking trading. Post-activation, the lock has no effect (one-way enable).

```solidity
function _transfer(address from, address to, uint256 amount) internal {
    ...
    require(
        tradingActive ||
            _isExcludedFromTradingLock[from] ||
            _isExcludedFromTradingLock[to],
        "Trading is not active"
    );
    ...
}
```

**Impact:**
- Pre-launch: Trading can be unintentionally or intentionally blocked; users may face failed transactions and gas loss.
- Misconfiguration risk: If LP added while `tradingActive == false` and pair/router not whitelisted, trading is impossible until fixed.

**Location:**
`_transfer()` require gate.

**üí° Recommendation:**
> **Action Required:** Operational safeguards:
> 1. Before adding liquidity, either enable `tradingActive` or whitelist router and pair addresses.
> 2. Announce trading enablement beforehand to avoid failed transactions.
> - Alternative: Consider two-way pause with timelock/governance if runtime control is desired.

#### üü¢ [L-1] ERC20 Allowance Race Condition (Standard Approval Issue)

**Description:**
Using `approve()` to change a non-zero allowance to another non-zero value can enable the well-known race condition where a spender front-runs the change.

```solidity
function approve(address spender, uint256 amount) external returns (bool) {
    _approve(msg.sender, spender, amount);
    return true;
}
```

**Impact:**
Potential double-spend by a malicious spender during allowance updates.

**Location:**
`approve()` and allowance logic.

**üí° Recommendation:**
> **Action Required:** User guidance and safer patterns:
> - Recommend setting allowance to zero before increasing, or use `increaseAllowance()/decreaseAllowance()`.
> - Optionally enforce zero-first in `approve()` (may break integrations).
